## 一、技术选型方案

### 1.1 技术栈总览

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| 语言 | Java | 25 (LTS) | 当前开发环境实际版本，长期支持 |
| 应用框架 | Spring Boot | 4.0.x | DI/IoC、事务管理、配置管理，轻量级；需验证与 JDK 25 兼容性 |
| UI 框架 | JavaFX | 24 (via OpenJFX) | 现代 UI、CSS 皮肤、FXML 布局、跨平台 |
| 数据库 | H2 | 2.2.x | 纯 Java 嵌入式、零安装、文件级存储、兼容 MySQL 模式 |
| ORM | Spring Data JPA + Hibernate | 6.x | 减少样板代码、自动建表、JPQL 查询 |
| 数据库连接池 | HikariCP | (Spring Boot 内置) | 高性能、轻量 |
| 构建工具 | Maven | 3.9+ | 依赖管理、插件丰富、社区活跃 |
| 打包工具 | jpackage | JDK 25 内置 | 生成原生安装包，集成 JRE，无需用户安装 Java |
| 报表导出 | Apache POI | 5.x | Excel 读写，P1 迭代使用 |
| 日志 | SLF4J + Logback | (Spring Boot 内置) | 标准日志方案 |

### 1.2 选型详解

#### 1.2.1 UI 框架：JavaFX

**为什么选 JavaFX 而非 Swing？**

| 对比维度 | JavaFX | Swing |
|----------|--------|-------|
| 外观风格 | 现代、CSS 可定制 | 传统、观感陈旧 |
| 布局方式 | FXML 声明式 + 代码式 | 纯代码式 |
| 动画支持 | 内置动画 API | 需自行实现 |
| 3D 支持 | 内置 | 不支持 |
| 社区维护 | OpenJFX 活跃维护 | 仅维护模式 |
| 与需求匹配 | "简约美观、有高级感" — 高度匹配 | 难以达到"高级感"要求 |

**JavaFX 集成 Spring Boot 方案**：采用 `AbstractJavaFxApplicationSupport` 模式，Spring Boot 管理 Bean 生命周期，JavaFX 负责 UI 渲染。

#### 1.2.2 数据库：H2

**为什么选 H2 而非 SQLite？**

| 对比维度 | H2 | SQLite |
|----------|-----|--------|
| Java 集成 | 纯 Java 实现，零原生依赖 | 需 JDBC 原生驱动（jni） |
| 打包友好 | 单个 jar 即可 | 需打包平台相关 .dll/.so |
| SQL 兼容 | 兼容 MySQL/PostgreSQL 模式 | 自有 SQL 方言 |
| 功能特性 | 支持存储过程、触发器、全文搜索 | 功能较弱 |
| 迁移便利 | 后续可平滑迁移到 MySQL | 方言差异大 |

H2 使用 **MySQL 兼容模式**，后续如需迁移到 Web 端 MySQL 数据库，几乎零改动。

#### 1.2.3 打包方案：jpackage

**打包流程**：

```
Maven 编译 → 生成 Fat JAR → jlink 裁剪 JRE → jpackage 生成 .exe/.msi 安装包
```

- `jlink`：根据模块依赖裁剪出最小 JRE（约 40-50MB），无需用户安装完整 JDK
- `jpackage`：将裁剪后的 JRE + 应用 JAR 打包为 Windows 安装包
- 用户安装后即可使用，真正的"开箱即用"

### 1.3 设计模式应用

| 设计模式 | 应用场景 | 说明 |
|----------|----------|------|
| **MVC** | UI 层整体架构 | Controller 处理事件、View(FXML) 渲染、Model(Entity) 数据 |
| **状态模式** | 订单生命周期 | OrderStatus 枚举驱动状态流转，Service 层校验状态转换合法性 |
| **策略模式** | 定价计算 | 不同包间类型（一口价/大厅按星期）对应不同计价策略 |
| **观察者模式** | 包间状态刷新 | 订单状态变更 → 通知 UI 刷新包间卡片 |
| **仓储模式** | 数据访问层 | Spring Data JPA Repository 封装数据访问 |
| **模板方法** | 结算流程 | 定义结算骨架，不同场景（有预约/无预约）填充具体步骤 |
| **建造者模式** | 订单创建 | 复杂订单对象构造，链式调用设置字段 |

### 2.3 各层职责与依赖

```
┌──────────────────────────────────────────────────────┐
│  依赖方向：UI → Service → Repository → Entity         │
│  原则：上层依赖下层，下层不感知上层                      │
│  跨层通信：Spring Event 实现 UI ↔ Service 解耦通知      │
└──────────────────────────────────────────────────────┘
```

| 层 | 职责 | 依赖 |
|----|------|------|
| **ui** | 界面渲染、用户交互、事件绑定 | service, dto, event |
| **service** | 业务逻辑、事务管理、状态流转 | repository, entity, enums |
| **repository** | 数据持久化、CRUD 操作 | entity |
| **entity** | 数据模型定义、JPA 映射 | 无（最底层） |
| **config** | Spring 配置、数据库初始化 | 横切所有层 |

### 2.4 订单状态机设计（核心）

```
                    ┌──────────────────┐
                    │    RESERVED      │
                    │    (预约中)       │
                    └───┬──────────┬───┘
                        │          │
              [到店开台] │          │ [爽约]
                        │          │
                        ▼          ▼
              ┌─────────────┐  ┌──────────────┐
              │ IN_PROGRESS │  │  CANCELLED   │
              │  (进行中)    │  │  (废订单)     │
              └──────┬──────┘  └──────────────┘
                     │
           [收款结算] │
                     │
                     ▼
              ┌─────────────┐
              │  FINISHED   │
              │  (已结束)    │
              └─────────────┘

无预约直接开台：  (无) → IN_PROGRESS → FINISHED
```

状态转换校验规则（在 OrderService 中实现）：

| 当前状态 | 允许转换到 | 触发操作 |
|----------|-----------|----------|
| (新建) | RESERVED | 创建预约订单 |
| (新建) | IN_PROGRESS | 直接开台 |
| RESERVED | IN_PROGRESS | 预约到店开台 |
| RESERVED | CANCELLED | 标记爽约 |
| IN_PROGRESS | FINISHED | 收款后闭台 |

### 2.5 定价策略（策略模式）

```
PricingStrategy (接口)
    ├── FixedPriceStrategy       # 固定价格：小包 228 / 中包 298 / 大包 398 / 大厅大包 398
    └── WeekdayPriceStrategy     # 按星期：大厅 周一~周四 35 / 周五~周日 45
```

PricingService 根据 RoomType 动态选择策略，新增包间类型或定价规则只需添加新策略类。

### 3.2 建表 DDL（H2 MySQL 兼容模式）

```sql
-- 设置 MySQL 兼容模式
SET MODE MySQL;

-- ==================== 包间表 ====================
CREATE TABLE IF NOT EXISTS room (
    id          BIGINT          AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(50)     NOT NULL COMMENT '包间名称',
    type        VARCHAR(20)     NOT NULL COMMENT '包间类型：SMALL/MEDIUM/LARGE/HALL/HALL_LARGE',
    price       DECIMAL(10,2)   NOT NULL DEFAULT 0 COMMENT '基础价格',
    status      VARCHAR(20)     NOT NULL DEFAULT 'IDLE' COMMENT '当前状态：IDLE/RESERVED/IN_USE',
    create_time DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '更新时间'
) COMMENT '包间表';

-- ==================== 订单表 ====================
CREATE TABLE IF NOT EXISTS `order` (
    id              BIGINT          AUTO_INCREMENT PRIMARY KEY,
    room_id         BIGINT          NOT NULL COMMENT '关联包间ID',
    reserve_time    DATETIME        NULL COMMENT '预约时间',
    start_time      DATETIME        NULL COMMENT '开台时间',
    end_time        DATETIME        NULL COMMENT '闭台时间',
    people_count    INT             NOT NULL DEFAULT 0 COMMENT '到店人数',
    deposit         DECIMAL(10,2)   NOT NULL DEFAULT 0 COMMENT '定金金额',
    is_deposit_paid TINYINT(1)      NOT NULL DEFAULT 0 COMMENT '是否定金支付',
    actual_payment  DECIMAL(10,2)   NOT NULL DEFAULT 0 COMMENT '实付款',
    refund          DECIMAL(10,2)   NOT NULL DEFAULT 0 COMMENT '退款金额',
    is_meituan      TINYINT(1)      NOT NULL DEFAULT 0 COMMENT '是否美团验券',
    overnight_count INT             NOT NULL DEFAULT 0 COMMENT '过夜人数',
    overnight_fee   DECIMAL(10,2)   NOT NULL DEFAULT 0 COMMENT '过夜费总额',
    status          VARCHAR(20)     NOT NULL DEFAULT 'RESERVED' COMMENT '订单状态：RESERVED/IN_PROGRESS/FINISHED/CANCELLED',
    create_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '最后修改时间',
    CONSTRAINT fk_order_room FOREIGN KEY (room_id) REFERENCES room(id),
    INDEX idx_order_status (status),
    INDEX idx_order_room_id (room_id),
    INDEX idx_order_create_time (create_time)
) COMMENT '订单表';

-- ==================== 操作日志表 ====================
CREATE TABLE IF NOT EXISTS audit_log (
    id          BIGINT          AUTO_INCREMENT PRIMARY KEY,
    order_id    BIGINT          NOT NULL COMMENT '关联订单ID',
    action      VARCHAR(50)     NOT NULL COMMENT '操作类型：CREATE/START/PAY/CLOSE/MODIFY/CANCEL',
    detail      TEXT            NULL COMMENT '操作详情',
    create_time DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '操作时间',
    CONSTRAINT fk_audit_order FOREIGN KEY (order_id) REFERENCES `order`(id),
    INDEX idx_audit_order_id (order_id),
    INDEX idx_audit_create_time (create_time)
) COMMENT '操作日志表';
```

## 五、扩展点设计

### 5.1 扩展原则

> 遵循 **开闭原则（OCP）**：对扩展开放，对修改关闭。

### 5.2 预留扩展点

#### 5.2.1 数据同步接口（Web/移动端扩展）

```
com.businessledger.sync/                    # 后续迭代新增模块
├── SyncService.java                        # 数据同步服务接口
├── SyncDataExporter.java                   # 数据导出（JSON/Protobuf）
├── SyncDataImporter.java                   # 数据导入
└── SyncConflictResolver.java               # 冲突解决策略
```

**设计要点**：
- 数据库使用 `AUTO_SERVER=TRUE` 模式，支持多进程访问
- 实体类实现 `Serializable`，支持序列化传输
- 订单表预留 `sync_version` 字段（乐观锁），用于多端冲突检测
- PricingService 接口化，后续可替换为远程调用

#### 5.2.2 定价策略扩展

```
PricingStrategy (接口)
    ├── FixedPriceStrategy          # 现有：固定价格
    ├── WeekdayPriceStrategy        # 现有：按星期计价
    └── (扩展) HolidayPriceStrategy  # 未来：节假日特殊定价
    └── (扩展) TimeBasedStrategy     # 未来：按时段差异化定价
```

只需新增策略类并注册到 `PricingService` 即可，无需修改现有代码。

## 七、附录

### 7.1 技术选型决策记录

| 决策项 | 候选方案 | 最终选择 | 决策理由 |
|--------|----------|----------|----------|
| JDK 版本 | 17 / 21 / 25 | **25 (LTS)** | 当前开发环境实际版本，最新 LTS |
| UI 框架 | JavaFX / Swing / Electron | **JavaFX** | 现代、CSS 可定制、原生性能 |
| 数据库 | H2 / SQLite / Derby | **H2** | 纯 Java、MySQL 兼容、无原生依赖 |
| DI 框架 | Spring Boot / Guice / 手动 | **Spring Boot** | 生态完整、事务管理、扩展性好 |
| ORM | JPA / MyBatis / JDBC | **JPA** | 简单 CRUD 无需手写 SQL、自动建表 |
| 打包 | jpackage / launch4j / exe4j | **jpackage** | JDK 原生、集成 JRE、无第三方依赖 |

### 7.2 关键设计理念

1. **避免过度设计**：v1.0 仅实现 P0 功能，P1/P2 预留扩展点但不实现
2. **向经典看齐**：严格遵循 MVC + Service + Repository 分层
3. **开箱即用**：安装包集成 JRE + H2，用户零环境配置
4. **数据安全**：数据库文件本地存储，支持手动备份/恢复
5. **操作可追溯**：AuditLog 记录所有订单变更，定长清理

### 7.3 迭代路线图

| 版本 | 内容 | 预计时间 |
|------|------|----------|
| v1.0 | P0 功能：台账管理 + 包间管理 + 支付管理 | 当前迭代 |
| v1.1 | P1 功能：报表导出 + 历史查询 + 数据备份/恢复 | 后续迭代 |
| v2.0 | Web 端开发 + 数据同步 | 远期规划 |
