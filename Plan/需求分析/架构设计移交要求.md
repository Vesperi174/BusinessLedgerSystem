# 架构设计移交要求

> 从产品经理移交给架构设计师，基于需求分析文档提炼。

---

## 一、项目概要

**BusinessLedgerSystem** —— 桌游店业务台账系统，桌面单机应用，仅店主一人使用。

## 二、核心约束

| 约束 | 说明 |
|------|------|
| 用户数 | 仅 1 人（店主），无需多用户/角色/权限体系 |
| 操作系统 | Windows 10，后续兼容 Windows 11 |
| 部署方式 | 安装包内集成 JRE + 数据库，开箱即用，用户电脑无 Java/Maven/DB 环境 |
| 数据库 | 轻量级嵌入式数据库（H2 或 SQLite），文件级存储 |
| 界面风格 | 简约美观、有高级感，不花哨不简陋 |
| 设计原则 | 遵循经典设计模式，向设计目标看齐，**避免过度设计** |
| 扩展预留 | 后续可能开发 Web/移动端做数据同步，架构需预留扩展点 |

## 三、功能模块

| 模块 | 优先级 | 核心职责 |
|------|--------|----------|
| 台账管理 | P0 | 订单全生命周期：创建预约 → 开台 → 收款结算 → 闭台 / 爽约 |
| 包间管理 | P0 | 7 个包间实时状态可视化，预约时校验时段可用性 |
| 支付管理 | P0 | 定金（默认50元可调）、实付款、美团验券标记、退款 |
| 报表导出 | P1 | 一键生成 Excel 财报（后续迭代） |

## 四、订单生命周期

```
                    ┌─ 有预约（定金默认50元）→ 开台 ─┐
创建预约 ────────────┤                                ├─→ 收款结算 → 闭台 → 已结束
                    └─ 爽约（未到）→ 废订单 ─────────┘

直接到店（无预约）→ 开台 → 收款结算 → 闭台 → 已结束
```

- 预约非必选，支持直接开台
- 订单创建后支持修改
- 预约时间精确到几点几分，可不填满

## 五、数据实体

### 订单（Order）
预约时间、开台时间、闭台时间、人数、包间（外键）、定金、是否定金支付、实付款、退款、是否美团验券、过夜人数、过夜费、订单状态（预约中/进行中/已结束/废订单）、创建时间、修改时间

### 包间（Room）
7 个包间：1号小包(228元)、2号中包(298)、3号大包(398)、4号中包(298)、5号大包(398)、大厅(35/45)、大厅大包(398)。状态：空闲/预约/使用。

### 操作日志（AuditLog）
记录订单增删改操作，定长清理。

## 六、关键业务规则

- **包间一口价**，不限时长；营业时间：包间 10:00~24:00，大厅 13:00~23:30
- **大厅**：周一至周四 35 元，周五至周日 45 元
- **过夜费**：20 元/人
- **定金**：默认 50 元，可调整，结算时退回
- **美团验券**：仅标记是/否
- **数据备份/恢复**：手动操作
- **操作日志**：定长清理

## 七、架构师需产出

1. **技术选型方案**：UI 框架、ORM/数据访问层、打包方案（jlink / jpackage / launch4j 等）
2. **系统分层架构**：模块划分、包结构、设计模式应用
3. **数据库设计**：ER 图、建表 DDL
4. **UI 布局方案**：主界面（包间状态可视化 + 订单列表）、预约/结算弹窗
5. **扩展点设计**：为后续 Web/移动端同步预留接口
