# 数据需求

## 数据模型概览

系统采用轻量级嵌入式数据库（如 H2 / SQLite），无需单独安装数据库服务。核心实体关系如下：

```
包间 1 ──── N 订单
```

## 核心实体

### 实体一：订单（Order）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 整数（主键） | 订单唯一标识 |
| room_id | 整数（外键） | 关联包间 |
| reserve_time | 日期时间 | 预约时间 |
| start_time | 日期时间 | 开台时间（可为空，未开台时为空） |
| end_time | 日期时间 | 闭台时间（可为空，未闭台时为空） |
| people_count | 整数 | 到店人数 |
| deposit | 金额 | 预约定金，默认 50 元，允许店主调整（无预约则为 0） |
| is_deposit_paid | 布尔 | 是否通过定金支付 |
| actual_payment | 金额 | 实付款金额 |
| refund | 金额 | 退款金额 |
| is_meituan | 布尔 | 是否美团验券 |
| status | 枚举 | 订单状态：预约中 / 进行中 / 已结束 / 废订单 |
| overnight_count | 整数 | 过夜人数（默认 0） |
| overnight_fee | 金额 | 过夜费（过夜人数 × 20 元） |
| create_time | 日期时间 | 记录创建时间 |
| update_time | 日期时间 | 最后修改时间 |

### 实体二：包间（Room）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 整数（主键） | 包间唯一标识 |
| name | 字符串 | 包间名称（如"1号小包"） |
| type | 枚举 | 包间类型：小包 / 中包 / 大包 / 大厅 / 大厅大包 |
| price | 金额 | 价格 |
| status | 枚举 | 当前状态：空闲 / 预约 / 使用 |

### 实体三：操作日志（AuditLog）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | 整数（主键） | 日志唯一标识 |
| order_id | 整数（外键） | 关联订单 |
| action | 字符串 | 操作类型（创建、开台、付款、闭台、修改等） |
| detail | 文本 | 操作详情 |
| create_time | 日期时间 | 操作时间 |

## 数据流

```
用户操作 → UI 界面 → 业务逻辑层 → 数据访问层 → 嵌入式数据库
                                              ↕
                                       备份/恢复（文件级别）
```

## 数据字典

- **订单状态**：`RESERVED`（预约中）、`IN_PROGRESS`（进行中）、`FINISHED`（已结束）、`CANCELLED`（废订单/爽约）
- **包间类型**：`SMALL`（小包）、`MEDIUM`（中包）、`LARGE`（大包）、`HALL`（大厅）、`HALL_LARGE`（大厅大包）
- **包间状态**：`IDLE`（空闲）、`RESERVED`（预约）、`IN_USE`（使用）

## 数据存储策略

- **备份/恢复**：支持手动备份数据库文件到指定路径，支持从备份文件恢复。
- **历史查询**：支持按日期范围、包间、订单状态等维度查询历史订单。
- **数据导出**：作为可扩展项（后续迭代）。
- **日志清理**：操作日志定长清理，保留最近 N 条记录。
